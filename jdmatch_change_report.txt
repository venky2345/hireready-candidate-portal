JD Match Fallback & Frontend Guard Change Report
================================================

Date: 2026-03-13

1) Files changed
----------------
- netlify/functions/generateResume.js
- jdmatch.html

2) Exact logic added/changed (backend)
-------------------------------------
- `generateResume.js` now returns structured fallback metadata:
  - `fallbackUsed`: boolean indicating that a deterministic fallback resume was built.
  - `fallbackReason`: short code describing why fallback/error occurred.
  - `message`: human-readable message suitable for direct display in the UI.
- `makeFallback()` was updated to:
  - Accept `(fallbackReason, message, extraMeta)` and log these values with `safeCandidateId`.
  - Return `ok: false` whenever `fallbackUsed === true`, so placeholder fallback output is never treated as a silent success.
  - Preserve `filename`, `resumeHtml`, `metrics`, `templateId`, `templateIdUsed`, and `templateFileUsed` for diagnostics.
- Additional backend error responses were enriched with `fallbackReason` and `message`:
  - Missing template mapping:
    - `fallbackReason: 'missing_template_mapping'`
    - `fallbackUsed: false`
  - Template read failure:
    - `fallbackReason: 'template_read_failed'`
    - Includes `templateId`, `templateIdUsed`, and `templateFileUsed` when available.
  - AI/model failures (timeout, HTTP error, non-OK status):
    - `fallbackReason: 'model_generation_failed'`
    - Returned via `makeFallback()` with `fallbackUsed: true` and `ok: false`.
  - Invalid AI JSON / missing fields:
    - `fallbackReason: 'generated_content_invalid'`
    - `fallbackUsed: false`
  - Unexpected internal error during template population:
    - `fallbackReason: 'template_population_failed'`
    - `fallbackUsed: false`
- Successful AI generations now explicitly include:
  - `fallbackUsed: false`
  - `fallbackReason: null`

3) Exact fallback conditions handled
------------------------------------
- `missing_template_mapping`:
  - Triggered when the requested `templateId` is not found in `JDM_TEMPLATE_MAP`.
  - Backend returns `ok: false`, `fallbackUsed: false`, and a clear `message`.
- `template_read_failed`:
  - Triggered when the HTML template file for the resolved id cannot be read from disk.
  - Backend returns `ok: false`, `fallbackUsed: false`, and includes template metadata when known.
- `model_generation_failed`:
  - Triggered when the OpenAI chat completion request:
    - Times out (AbortError after ~8s), or
    - Fails at the HTTP/request layer, or
    - Returns a non-OK HTTP status.
  - Backend builds a deterministic fallback resume but now marks:
    - `ok: false`
    - `fallbackUsed: true`
    - `fallbackReason: 'model_generation_failed'`
    - `message` describing the timeout or model failure.
- `generated_content_invalid`:
  - Triggered when:
    - The OpenAI response body cannot be parsed as JSON, or
    - The parsed JSON is missing required keys such as `filename`, `replacements`, or `addedSkills`.
  - Backend returns `ok: false`, `fallbackUsed: false`, and a descriptive `message`.
- `template_population_failed`:
  - Triggered by unexpected errors in the tail end of `generateResume` after a valid AI response.
  - Backend returns `ok: false`, `fallbackUsed: false`, and a generic internal-error `message`.

4) Exact frontend guards added (jdmatch.html)
---------------------------------------------
- `handleGenerate()` (button-based generation flow):
  - After a 200 HTTP response and basic template-id checks, a hard failure guard now runs before assigning `currentGeneration`:
    - Treats generation as failed if:
      - `data.fallbackUsed === true`, OR
      - `data.ok === false`, OR
      - `data.filename` starts with `fallback_`, OR
      - `data.resumeHtml` is missing or trimmed length is below the existing threshold (200 characters).
  - On any of these conditions:
    - Logs a structured console error including `fallbackReason`, `message`, and HTML length.
    - Clears `currentGeneration` and hides the `outputSection`.
    - Shows an error toast using `data.message` or `data.error` when present, otherwise a generic failure message.
  - For non-OK HTTP responses or `ok: false` JSON payloads:
    - Error handling now logs `fallbackReason` and `message`.
    - Throws an Error whose message includes both the reason code and the message text, so the user-visible toast explains what went wrong.
- `handleGenerateSubmit(event)` (form submission flow):
  - Mirrors the same guard and error-propagation behavior as `handleGenerate()`:
    - Rejects responses where `fallbackUsed` is true, `ok` is false, the filename starts with `fallback_`, or `resumeHtml` is missing/too short.
    - Clears `currentGeneration`, hides `outputSection`, and shows an error toast based on backend `message`/`error`.
  - Backend failures with `ok: false` now surface `fallbackReason` and `message` in both console logs and the thrown error string.

5) Exact user-visible behavior after the fix
--------------------------------------------
- On successful AI-backed generation:
  - The preview uses the real selected template, consistent with both `templateIdSent` and `templateIdUsed`.
  - Downloads (PDF/Word/HTML) are enabled only when `currentGeneration.resumeHtml` passes existing length checks.
  - Filenames do not start with `fallback_`, and no fallback flags are present.
  - A success toast is shown, and history entries reflect genuine generated resumes.
- On backend fallback or failure (including timeouts, model errors, invalid AI JSON, missing template mapping, or template read failures):
  - The backend returns `ok: false` with `fallbackReason` and `message`.
  - The jdmatch frontend:
    - Does not set `currentGeneration` to the placeholder/fallback output.
    - Hides or leaves hidden the `outputSection` so preview and download actions cannot proceed.
    - Does not show any success toast.
    - Shows a clear error toast derived from the backend `message` and/or `fallbackReason` so users understand why generation failed.

4) Root cause (Progress overlay null classList; function prompt too large including full template HTML; no OpenAI timeout; response.json on HTML)
showPhasedProgress/hidePhasedProgress assumed DOM elements existed and called classList directly. The Netlify function built a giant prompt containing entire template HTML plus JD and resume which frequently triggered OpenAI latency and Netlify 504; when timeout occurred the frontend attempted response.json() on an HTML page and crashed. The function also lacked a global JSON response helper and returned HTML on errors.

- Modified showPhasedProgress to bail if overlay missing, guard phase elements, progressBar and percentage, and always return a Promise.
- Modified hidePhasedProgress to re-query DOM inside setTimeout, guard all elements, and wrap in try/catch.
- Wrapped showPhasedProgress call in handleGenerate in try/catch; wrapped hidePhasedProgress calls in try/catch.
- Rewrote handleGenerate response handling: read raw text, inspect content-type, parse JSON conditionally with error logging/toast, handle non-ok statuses, always hide progress and restore button in finally.
- Added comprehensive respond() helper to Netlify function; enforced POST only; safe JSON.parse of request body; compatibility for OPENAI_API_KEY/API_KEY; trimmed and limited jd, resume, and mustIncludeSkills lengths; extract placeholder keys from template and build user prompt without full HTML; added AbortController with 18s timeout; reduced temperature to 0.4 and max_tokens to 900; always return JSON on every path including timeouts and errors; construct resumeHtml server-side from replacements; clamp metrics; log relevant request details.

   ----------------------------------------------------------------------

     of whether the template uses native section containers or a flat layout.

       - Single-column templates: one group of all sections → reordered as before.

    - `[generateResume] AI content JSON parse success` on success.
    - A detailed error and falling back to deterministic generation on failure.
  - When the AI response is missing required fields (`filename`, `replacements`, `addedSkills`):
    - Logs `[generateResume] AI response structure invalid, attempting deterministic fallback`.
    - Attempts deterministic fallback before returning `ok: false`.

-------------------------------------
- Reused and relied on `buildDeterministicResume(options)` instead of hard-failing on AI parse/structure errors:
  - When envelope parse fails, content parse fails, or structure is invalid, the backend now:

    - Applies parsed sections into the same template placeholders used for AI mode.
    - Returns `ok: true` only if the resulting HTML passes backend semantic validation via `validateRenderedResumeHtml`.
  - Deterministic fallback is now invoked for:
    - Envelope JSON parse failures.
    - AI content JSON parse failures.
    - AI responses that are missing required keys.

5) Final validation guarantees
------------------------------
- `validateRenderedResumeHtml` is used for both AI and deterministic results to ensure:
  - No unresolved `{{PLACEHOLDER}}` tokens remain.
  - No obvious placeholder/sample text like "CANDIDATE NAME", "PROFESSIONAL SUMMARY", "CORE PROFESSIONAL COMPETENCIES", "MM/YYYY", or "ORGANIZATION NAME" appears in the final HTML.
  - The HTML is not merely a large raw text blob with minimal structural tags.
- If validation fails after:
  - AI rendering, and
  - Deterministic fallback attempts,
  then the backend:
  - Returns `ok: false` with a clear `error`/`message`.
  - Does **not** return malformed `resumeHtml` as success.

   Fix three critical issues:

   ISSUE 2: PDF and DOCX downloads are creating blank files when resumeHtml is empty
   ISSUE 3: No visual confirmation of which template was actually used in generation

4) Root cause

   ROOT CAUSE 1 — Preview used unvalidated HTML:
   previewResume() checked for missing HTML but did not validate minimum length
   or detect blank content (HTML with only whitespace/empty tags). This allowed
   preview modal to open with effectively blank content.

   ROOT CAUSE 2 — Downloads never validated content:
   downloadPDF(), downloadDOC(), downloadHTML() had no validation beyond checking
   if currentGeneration exists. They never verified that resumeHtml:
   - is not empty string
   - has sufficient length (not just a few chars)
   - contains actual visible text content (not just empty HTML wrapper)
   Result: blank PDF/DOCX files were generated when:
   - backend returned minimal fallback HTML
   - resumeHtml was accidentally overwritten with empty string
   - generation succeeded but with empty template render

   ROOT CAUSE 3 — No visual feedback of generated template:
   After generation, user had no way to verify which template was actually used.
   Output section only showed metrics and download buttons. If template mismatch
   occurred (selected t02 but backend used t01), user would not notice until
   downloading and opening the file.
